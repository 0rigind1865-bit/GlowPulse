# GlowPulse 專案架構說明

GlowPulse 是 GlowMoment 的社群自動化工具，負責在 Threads 上自動發布產品宣傳貼文、分析他人貼文是否為潛在客戶自動產出回覆，並定期產出發文成效週報與 AI 改進建議。

---

## 目錄結構

```

│   ├── data/               # 純資料層，不含任何邏輯

│   ├── utils/

│   ├── types/
│   │   └── opencc-js.d.ts  # opencc-js 型別宣告（社群套件缺少官方型別）

├── docs/
│   ├── 專案架構說明.md       # 本文件（每日結束時更新，執行 /update-arch 指令）
│   ├── reference-posts.md  # 參考貼文庫（手動維護，供未來 analyzeReference 任務使用）
│   └── reports/            # 週報輸出目錄（YYYY-MM-DD.md，自動產生）
├── .claude/
│   └── commands/
│       └── update-arch.md  # Claude Code 自訂指令：檢查今日變更並更新本文件

├── package.json
└── tsconfig.json
```

## 各層職責與設計原則

### `src/data/` — 產品知識庫

**沒有任何 import，只有型別定義與 `export const`。**

這一層是最常被修改的地方，但修改風險最低，因為它不含邏輯：

| 檔案 | 修改時機 |
|---|---|
| `brand.ts` | 調整品牌語氣、寫作禁忌、字數限制；含去 AI 化寫作風格指令（5 條具體規則）；週報 AI 可自動更新寫作原則區塊 |
| `features.ts` | 新增功能、修改某功能的痛點描述或切入角度；`imagePost.ts` 遇到未知截圖時會自動呼叫 HF 生成新功能並寫入此檔 |
| `screenshots.ts` | 新增截圖時手動加入記錄；`featureName` 與 `description` 首次為空時由 `imagePost.ts` 自動回寫 |
| `styles.ts` | 新增發文風格、調整現有風格的指令說明；`weeklyReport` 與 `analyzeReference` 皆可自動更新 |

### `src/utils/` — 跨任務共用工具

- **`dataUpdater.ts`**：`weeklyReport.ts` 與 `analyzeReference.ts` 共用的 data 層更新邏輯
  - `parseDataUpdates(aiOutput)` → 從 AI 輸出中提取 `<<<UPDATES_JSON>>>` 區塊並解析
  - `stripUpdatesBlock(aiOutput)` → 移除 JSON 區塊，只保留 Markdown 分析內容
  - `updateBrandTs(principles)` → 寫回 `brand.ts` 的【寫作原則】區塊
  - `updateStylesTs(styles)` → 重新生成 `styles.ts` 的 `POST_STYLES` 陣列
  - `applyDataUpdates(updates)` → 依序呼叫上兩個函式，回傳執行 log

### `src/services/` — 外部 API 封裝

只處理 HTTP 層，所有錯誤一律轉為 `Error` 向上拋出。不判斷業務邏輯。

- **`hf.ts`**：封裝兩支 Hugging Face 端點；底層整合 `opencc-js` 自動將輸出從簡體轉為繁體中文
  - `callChatCompletion()` → `/v1/chat/completions`（發文生成 / 回覆建議）
  - `callZeroShot()` → `/hf-inference/models/:id`（意圖分類）
  - `normalizeTraditionalChinese()` → 內部工具函式，HF 輸出統一轉繁體（保底機制）
- **`claude.ts`**：封裝 Anthropic Claude Haiku
  - `callClaude(system, prompt, maxTokens)` → 週報分析，串流輸出
  - Claude 無餘額時 weeklyReport 自動退回 HF
- **`gemini.ts`**：封裝 Google Gemini 視覺 API（免費 tier）
  - `describeImageWithGemini(imageUrl)` → 下載圖片 → base64 → 回傳繁體中文畫面描述
  - 模型輪替：`gemini-2.5-flash` → `gemini-flash-latest` → `gemini-2.5-pro`（依序嘗試至成功）
  - `GeminiServiceError` → 結構化錯誤類別，含 `QUOTA_EXCEEDED` / `MODEL_NOT_FOUND` / `REQUEST_FAILED`
  - 含繁體中文驗證：若回傳夾雜簡體，自動要求模型改寫
- **`threads.ts`**：封裝 Threads Graph API
  - `refreshToken()` → 刷新長效 Token 並自動寫回 `.env`
  - `createContainer()` → 建立文字媒體容器（貼文尚未公開）
  - `createImageContainer(imageUrl, caption, token)` → 建立圖片媒體容器（使用公開 URL）
  - `publishContainer()` → 發布容器為公開貼文
  - `fetchMyPosts()` → 取得自己帳號過去 N 天的貼文列表
  - `fetchPostInsights()` → 取得單篇貼文的互動數據與加權分數

### `src/tasks/` — 業務流程

組合 `data/` 與 `services/`，但**不含 top-level 副作用**（不會在被 import 時自動執行）。

- **`autoPost.ts`**：`runAutoPost()`
  1. 依「今年第幾天」決定今日功能與風格（8 功能 × 3 風格 = 24 天一循環）
  2. 呼叫 HF Chat Completion 生成貼文
  3. 刷新 Token → 建立容器 → 發布
- **`analyzeReply.ts`**：`analyzeAndReply(postContent)`
  1. 零樣本分類判斷意圖（預約需求 / 美容抱怨 / 日常生活 / 商業廣告）
  2. 只有前兩類才視為潛在客戶
  3. 對潛在客戶產出 50 字以內的品牌回覆
- **`imagePost.ts`**：`runImagePost()`
  1. 依「今年第幾天」從 `screenshots.ts` 輪替選出今日截圖與風格
  2. **首次**（`featureName` 與 `description` 皆空）：Gemini 分析截圖 → HF 生成 Feature JSON → 自動回寫 `features.ts` 與 `screenshots.ts`
  3. **後續**（已有快取）：直接使用 `screenshots.ts` 的既有描述，不重跑 Gemini
  4. `analyzeVisualDescription()` → HF 將畫面描述轉成「畫面事實 / 使用者價值 / 發文切角」三行結構
  5. `generateCaption()` → 結合結構化重點 + 功能資訊 + 風格指令 → HF 生成圖片搭配文案
  6. 互動確認（輸入 y）→ 刷新 Token → 建立圖片容器（GitHub raw URL）→ 發布
  - ⚠️ Gemini 失敗時直接中止（不降級），避免圖文不符
- **`analyzeReference.ts`**：`runAnalyzeReference()`
  1. 讀取 `docs/reference-posts.md`，移除 HTML 注解與 code block，解析出真實貼文
  2. 依風格分類分組顯示貼文數量
  3. AI 分析（Claude 優先，無餘額退回 HF）從五個維度提取寫作技巧：開頭鉤子、懸念設計、句子節奏、情緒觸發、結尾設計
  4. 自動寫回 `brand.ts`（寫作原則）與 `styles.ts`（風格指令）
  5. 在 console 輸出完整分析報告供參考
  - 與 `weeklyReport` 互補：前者學外部優質貼文，後者學自身成效數據
- **`weeklyReport.ts`**：`runWeeklyReport()`
  1. 取得過去 7 天貼文，逐篇取得互動數據
  2. 依 engagementScore 排序，取高低互動各 5 篇送 AI
  3. AI 輸出 Markdown 分析 + `<<<UPDATES_JSON>>>` 結構化更新區塊
  4. 透過 `utils/dataUpdater.ts` 解析 JSON，寫回 `brand.ts` / `styles.ts`
  5. 儲存週報至 `docs/reports/YYYY-MM-DD.md`

### `src/agent.ts` — 協調器（唯一有副作用的入口）

讀取 `process.argv` 參數，分派到對應的 task：

| CLI 參數 | 執行內容 |
|---|---|
| `--post` | 執行 `runAutoPost()` |
| `--image-post` | 執行 `runImagePost()`，截圖視覺分析 → 生成文案 → 發布圖片貼文 |
| `--analyze "貼文內容"` | 執行 `analyzeAndReply()` |
| `--all` | 依序執行發文 + 分析示範貼文 |
| `--analyze-reference` | 執行 `runAnalyzeReference()`，解析參考貼文庫並自動更新 data 層 |
| `--report` | 執行 `runWeeklyReport()`，產生週報並自動更新 data 層 |
| （無參數） | 顯示使用說明 |

**新增任務的方式**：在 `tasks/` 建新檔案，export 一個函式，在 `agent.ts` 的 switch 中加一個 case。其他檔案不需要動。

## 環境變數

在 `.env` 設定以下變數：

| 變數名稱 | 說明 | 取得方式 |
|---|---|---|
| `HF_TOKEN` | Hugging Face Access Token | [huggingface.co/settings/tokens](https://huggingface.co/settings/tokens) |
| `ANTHROPIC_API_KEY` | Anthropic Claude API Key（週報分析用） | [console.anthropic.com](https://console.anthropic.com/) |
| `GEMINI_API_KEY` | Google Gemini API Key（截圖視覺描述用，免費 tier） | [aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey)，格式為 `AIza...` |
| `THREADS_ACCESS_TOKEN` | Threads 長效 Access Token（60 天） | Meta Developer → App → Threads API |
| `THREADS_USER_ID` | 你的 Threads 用戶 ID | `curl "https://graph.threads.net/v1.0/me?fields=id&access_token=你的Token"` |

### Token 注意事項

Threads 預設給**短效 Token**（幾小時過期），必須先換成**長效 Token**（60 天）：

```bash
# 用短效 Token 換取長效 Token（需要 App ID 與 App Secret）
curl "https://graph.threads.net/access_token?grant_type=th_exchange_token&client_id=APP_ID&client_secret=APP_SECRET&access_token=短效Token"
```

長效 Token 每次執行 `npm run post` 時會**自動刷新並寫回 `.env`**，只要 60 天內執行過一次就不會過期。

## 發文輪替邏輯

發文內容由「今年第幾天」決定，不隨機，每天結果一致（可重現）：

- **功能**：8 個，以 `dayOfYear % 8` 輪替
- **風格**：3 種，以 `dayOfYear % 3` 輪替
- **循環長度**：LCM(8, 3) = 24 天後才會出現完全相同的組合

若想強制指定功能或風格，可修改 `src/tasks/autoPost.ts` 中的 `getTodaysConfig()`。

## 週報流程說明（`weeklyReport.ts`）

```
刷新 Token
    ↓
fetchMyPosts()      取得過去 7 天的貼文列表
    ↓
fetchPostInsights() 逐篇取得互動數據（views / likes / replies / reposts / quotes）
    ↓
依 engagementScore 排序（加權：回覆×4、轉發×5、按讚×3）
    ↓
AI 分析（Claude Haiku 優先，無餘額退回 HF Qwen）
    ↓
解析 <<<UPDATES_JSON>>> 區塊 → 自動寫回 brand.ts / styles.ts
    ↓
輸出三個區塊：高互動特徵 / 低互動問題 / data 層具體修改建議
    ↓
儲存至 docs/reports/YYYY-MM-DD.md
```

## 日常使用節奏（建議）

| 頻率 | 指令 | 說明 |
|---|---|---|
| 每天 | `npm run post` 或 `npm run image-post` | 發文 |
| 每天（有新收藏時）| `npm run learn` | 從外部優質貼文學習寫作技巧 |
| 每週 | `npm run report` | 從自身互動數據反省什麼有效 |

`learn` 與 `report` 互補，共同驅動 data 層的持續優化。
