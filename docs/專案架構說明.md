# GlowPulse 專案架構說明

GlowPulse 是 GlowMoment 的社群自動化工具，負責在 Threads 上自動發布產品宣傳貼文（文字或圖片）、分析並直接回覆他人貼文中的潛在客戶、掃描自己貼文的留言批次回覆，並定期產出發文成效週報與 AI 改進建議。

---

## 目錄結構

```

│   │   ├── features.ts     # GlowMoment 功能的痛點/解法/切入角度（imagePost 自動擴充）
│   │   ├── models.ts       # 模型路由設定中心：task-based candidates 陣列，依序嘗試至第一個成功

│   ├── utils/
│   │   ├── confirm.ts      # 共用互動確認：confirmAction（y/n）+ promptChoice（多選）
│   │   ├── publish.ts      # 共用溢位處理：貼文超字數時詢問重新生成 / 留言接續 / 取消
│   │   └── dataUpdater.ts  # 共用工具：解析 <<<UPDATES_JSON>>>，寫回 brand.ts / styles.ts

│   └── agent.ts            # 協調器：讀 CLI 參數，決定執行哪個任務
```

## 各層職責與設計原則

### `src/data/` — 產品知識庫

**沒有任何 import，只有型別定義與 `export const`。**

這一層是最常被修改的地方，但修改風險最低，因為它不含邏輯：

| 檔案 | 修改時機 |
|---|---|
| `brand.ts` | 調整品牌語氣、寫作禁忌；含 Hashtag 固定清單（美業類 / 受眾身份 / 產品類）與【誘導留言格式】（問句型 / 二擇一 / 完形填空 / 投票感 / 資訊缺口）；`【從優質貼文學到的技巧】` 區塊由 `weeklyReport` / `analyzeReference` AI 自動更新，格式：技巧 + 為什麼有效 + 可直接套用的例句 |
| `features.ts` | 新增功能、修改某功能的痛點描述或切入角度；`imagePost.ts` 遇到未知截圖時會自動呼叫 AI 生成新功能並寫入此檔 |
| `models.ts` | 切換任何任務使用的 AI 模型；採 task-based **candidates 陣列**，每個任務各自設定優先順序，第一個成功的模型即回傳；`gemini.visionCandidates` 另外管理視覺模型輪替 |
| `screenshots.ts` | 新增截圖時手動加入記錄（檔名須為純 ASCII，Meta API 不接受中文/空格 URL）；`featureName` 與 `description` 首次為空時由 `imagePost.ts` 自動回寫 |
| `styles.ts` | 現有 8 種發文風格；`weeklyReport` / `analyzeReference` 可更新現有風格指令，`genStyles` 可從 brand.ts 技巧自動衍生並新增風格 |

### `src/utils/` — 跨任務共用工具

- **`confirm.ts`**：所有需要人工確認的互動提示
  - `confirmAction(message, preview?)` → 等待使用者輸入 `y` / `yes`；其他任何輸入都視為取消
  - `promptChoice(message, choices[])` → 顯示編號選項，回傳 1-based 選擇編號；適用超字數處理等多選情境
- **`publish.ts`**：`autoPost` 與 `imagePost` 共用的貼文溢位處理
  - `handleOverflow(text, regenerateFn, maxChars?)` → 若文字超過字數上限，詢問使用者：\[1\] 重新生成較短版本、\[2\] 以第一則留言接續發布、\[3\] 取消；未超限時直接回傳不詢問
- **`dataUpdater.ts`**：`weeklyReport.ts` 與 `analyzeReference.ts` 共用的 data 層更新邏輯
  - `parseDataUpdates(aiOutput)` → 從 AI 輸出中提取 `<<<UPDATES_JSON>>>` 區塊並解析
  - `stripUpdatesBlock(aiOutput)` → 移除 JSON 區塊，只保留 Markdown 分析內容
  - `updateBrandTs(incomingPrinciples)` → **累積合併模式**（非覆寫）：讀取現有技巧，以前 20 字元去重後補入 AI 新觀察；超過 `MAX_PRINCIPLES`（20 條）時以 FIFO 捨棄最舊條目，避免無限增長
  - `updateStylesTs(styles)` → 重新生成 `styles.ts` 的 `POST_STYLES` 陣列（字串自動去除換行，避免語法錯誤）
  - `applyDataUpdates(updates)` → 依序呼叫上兩個函式，回傳執行 log
  - *(私有)* `extractExistingPrinciples(content)` → 從 brand.ts 原始內容解析現有技巧清單
  - *(私有)* `mergePrinciples(existing, incoming)` → 去重合併，應用上限截斷

### `src/services/` — 外部 API 封裝

只處理 HTTP 層，所有錯誤一律轉為 `Error` 向上拋出。不判斷業務邏輯。

- **`generate.ts`**：統一 AI 生成介面（tasks 唯一需要呼叫的入口）
  - `generate(taskName, system, prompt, maxTokens?, temperature?)` → 從 `models.ts` 讀取 `tasks[taskName].candidates`，依序嘗試，第一個成功即回傳；HF 候選自動前置繁體中文指令
  - tasks 不再直接呼叫 `hf.ts` / `claude.ts` / `gemini.ts`（零樣本分類與視覺描述除外）
- **`hf.ts`**：封裝 Hugging Face 端點；底層整合 `opencc-js` 自動將輸出從簡體轉為繁體中文
  - `callChatCompletion()` → `/v1/chat/completions`（由 `generate.ts` 路由呼叫）
  - `callZeroShot()` → `/hf-inference/models/:id`（意圖分類，直接由 analyzeReply / replyTo / engage 呼叫）
- **`claude.ts`**：封裝 Anthropic Claude
  - `callClaude(system, prompt, maxTokens, model?)` → 串流輸出（由 `generate.ts` 路由呼叫）
- **`gemini.ts`**：封裝 Google Gemini API（免費 tier）
  - `describeImageWithGemini(imageUrl)` → 下載圖片 → base64 → 回傳繁體中文畫面描述（直接由 imagePost 呼叫）
  - `generateTextWithGemini(model, system, prompt)` → 純文字生成（由 `generate.ts` 路由呼叫）
  - 視覺模型輪替：依序嘗試 `MODELS.gemini.visionCandidates`（`gemini-2.5-flash` → `gemini-flash-latest` → `gemini-2.5-pro`）
  - `GeminiServiceError` → 結構化錯誤類別，含 `QUOTA_EXCEEDED` / `MODEL_NOT_FOUND` / `REQUEST_FAILED`
- **`threads.ts`**：封裝 Threads Graph API
  - `getUsableToken()` → 每次執行顯示 Token 剩餘天數；剩餘 ≤5 天自動刷新；`THREADS_TOKEN_REFRESHED_AT` 未設定時在 GitHub Actions 中輸出 `::warning::`
  - `refreshToken()` → 刷新長效 Token 並自動寫回 `.env`（GitHub Actions 環境無法寫回 Secrets，刷新後需手動更新）
  - `createContainer()` / `createImageContainer()` → 建立文字 / 圖片媒體容器
  - `createReplyContainer(text, replyToId, token)` → 建立留言容器（指定 `reply_to_id`）
  - `publishContainer()` → 發布容器為公開貼文或留言
  - `getContainerStatus()` → 查詢容器處理狀態（`IN_PROGRESS` / `FINISHED` / `PUBLISHED` / `ERROR`）
  - `waitForContainerReady()` → 輪詢容器狀態至終態，解決 reply container 異步驗證問題
  - `waitForPostReady()` → 輪詢貼文 ID 直到可被 API 查詢，確保 reply_to_id 有效
  - `splitForThread(text, maxChars)` → 在自然斷點分割長文（段落 → 句尾 → 換行 → 硬截斷）
  - `parseThreadsPostId(input)` → 解析 URL / shortcode / 數字 ID，Base64URL 解碼 shortcode
  - `getPost(postId, token)` → 取得指定貼文內容與作者（讀取公開貼文）
  - `getPostReplies(postId, token)` → 取得貼文的直接留言列表（需 `threads_manage_replies` 權限）
  - `fetchMyPosts()` → 取得自己帳號過去 N 天的貼文列表
  - `fetchPostInsights()` → 取得單篇貼文的互動數據與加權分數

### `src/tasks/` — 業務流程

組合 `data/` 與 `services/`，但**不含 top-level 副作用**（不會在被 import 時自動執行）。

- **`autoPost.ts`**：`runAutoPost(maxChars?)`
  1. 依「今年第幾天」決定今日功能與風格（8 功能 × 3 風格 = 24 天一循環）
  2. 呼叫 `generate()` 生成貼文（candidates 路由）
  3. `handleOverflow()` 處理超字數：詢問重新生成 / 留言接續 / 取消
  4. 刷新 Token → 建立容器 → 發布；若有接續留言，等主貼文就緒後發布留言
- **`analyzeReply.ts`**：`analyzeAndReply(postContent)`（僅產出文案，不發布）
  1. 零樣本分類判斷意圖（預約需求 / 美容抱怨 / 日常生活 / 商業廣告）
  2. 只有前兩類才視為潛在客戶
  3. 對潛在客戶產出 50 字以內的品牌回覆
- **`replyTo.ts`**：`runReplyTo(urlOrId)`
  1. `parseThreadsPostId()` 解析 URL shortcode 或數字 ID
  2. `getPost()` 自動取得貼文內容；API 無法讀取時改由使用者手動貼上
  3. 零樣本分類 → 非潛在客戶則略過
  4. 生成回覆 → 確認 → `waitForPostReady` → `createReplyContainer` → `waitForContainerReady` → 發布
- **`engage.ts`**：`runEngage(scanDays?)`
  1. `fetchMyPosts()` 取得最近 N 天（預設 7 天）自己的貼文
  2. 對每篇貼文呼叫 `getPostReplies()` 取得留言（需 `threads_manage_replies` 權限）
  3. 零樣本分類每則留言，找出潛在客戶
  4. 逐一顯示建議回覆，確認後發布；失敗個別記錄不中止整體流程
- **`imagePost.ts`**：`runImagePost(maxChars?, useLatest?)`
  1. **隨機**從 `screenshots.ts` 選出截圖（`--latest` 時選最後一筆）；風格同樣隨機
  2. 早期驗證：截圖 URL 含非 ASCII 字元立即中止並說明重新命名步驟
  3. **首次**（`featureName` / `description` 皆空）：Gemini 分析截圖 → AI 生成 Feature JSON → 回寫 `features.ts` 與 `screenshots.ts`
  4. **後續**（已有快取）：直接使用既有描述，不重跑 Gemini
  5. `generateCaption()` → 生成圖片搭配文案；`handleOverflow()` 處理超字數
  6. 互動確認 → 建立圖片容器 → 發布；接續留言同 autoPost 流程
  - ⚠️ Gemini 視覺分析失敗時直接中止（不降級），避免圖文不符
- **`analyzeReference.ts`**：`runAnalyzeReference()`
  1. 讀取 `docs/reference-posts.md`，移除 HTML 注解與 code block，解析出真實貼文
  2. 依風格分類分組顯示貼文數量
  3. AI 分析（透過 `generate()` 路由）從五個維度提取寫作技巧：開頭鉤子、懸念設計、句子節奏、情緒觸發、結尾設計
  4. 自動寫回 `brand.ts`（寫作原則）與 `styles.ts`（風格指令）
  5. 在 console 輸出完整分析報告供參考
  - 與 `weeklyReport` 互補：前者學外部優質貼文，後者學自身成效數據
- **`genStyles.ts`**：`runGenStyles()`
  1. 從 `BRAND_CONTEXT` 提取【從優質貼文學到的技巧】區塊
  2. 讀取現有 `POST_STYLES`，告知 AI 已有哪些風格避免重複
  3. AI（Claude → Gemini → HF）只輸出「純新增」的 `PostStyle` 條目（現有風格由 prompt 禁止輸出）
  4. 以名稱 Set 去重，過濾掉 AI 仍輸出的同名條目
  5. 代碼以 `[...POST_STYLES, ...genuinelyNew]` 合併，確保舊有風格完全不動
  6. 顯示新風格清單與設計說明，確認後以 `updateStylesTs(merged)` 寫回 `styles.ts`
- **`weeklyReport.ts`**：`runWeeklyReport()`
  1. 取得過去 7 天貼文，逐篇取得互動數據
  2. **樣本數門檻（`MIN_POSTS_FOR_AI_ANALYSIS = 5`）**：貼文數不足時跳過 AI 分析與 data 層更新，僅儲存純數據快照報告，避免小樣本覆蓋多週累積的技巧
  3. 依 engagementScore 排序，取高低互動各 5 篇送 AI
  4. AI 輸出 Markdown 分析 + `<<<UPDATES_JSON>>>` 結構化更新區塊
  5. 透過 `utils/dataUpdater.ts` 解析 JSON，以**累積合併**模式寫回 `brand.ts`（只補新技巧，不覆寫），整批更新 `styles.ts`
  6. 儲存週報至 `docs/reports/YYYY-MM-DD.md`

### `src/agent.ts` — 協調器（唯一有副作用的入口）

讀取 `process.argv` 參數，分派到對應的 task：

| CLI 參數 | 選項 | 執行內容 |
|---|---|---|
| `--post` | `--max-chars N` | 執行 `runAutoPost()`；超字數時詢問重新生成或留言接續 |
| `--image-post` | `--max-chars N`、`--latest` | 執行 `runImagePost()`；`--latest` 選最新截圖 |
| `--analyze "貼文內容"` | — | 執行 `analyzeAndReply()`，只產出回覆建議，不發布 |
| `--reply-to "URL或ID"` | — | 執行 `runReplyTo()`，分析指定貼文並直接回覆到 Threads |
| `--engage` | — | 執行 `runEngage()`，掃描自己近期貼文留言，批次回覆潛在客戶 |
| `--all` | — | 依序執行發文 + 分析示範貼文 |
| `--analyze-reference` | — | 執行 `runAnalyzeReference()`，解析參考貼文庫並自動更新 data 層 |
| `--gen-styles` | — | 執行 `runGenStyles()`，從 brand.ts 技巧自動衍生新風格並寫回 styles.ts |
| `--report` | — | 執行 `runWeeklyReport()`，產生週報並自動更新 data 層 |
| （無參數） | — | 顯示使用說明 |

**新增任務的方式**：在 `tasks/` 建新檔案，export 一個函式，在 `agent.ts` 的 switch 中加一個 case。其他檔案不需要動。

```
    │       ├── services/generate.ts → AI 衍生新 PostStyle 條目        │
    │       └── utils/dataUpdater.ts → updateStylesTs() 寫回 ─────────┤
    │                                                                   │
    └─── tasks/weeklyReport.ts                                         │
            ├── services/threads.ts  → 取得貼文列表 + 互動數據         │

                    ├── data/brand.ts   ← 自動寫回（寫作原則）
                    └── data/styles.ts ← 自動寫回（風格指令）

services/generate.ts（AI 路由層）
    └── data/models.ts → tasks[taskName].candidates
            ├── provider: 'hf'     → services/hf.ts
            ├── provider: 'claude' → services/claude.ts
```

## 環境變數

在 `.env` 設定以下變數：

| 變數名稱 | 說明 | 取得方式 |
|---|---|---|
| `HF_TOKEN` | Hugging Face Access Token | [huggingface.co/settings/tokens](https://huggingface.co/settings/tokens) |
| `ANTHROPIC_API_KEY` | Anthropic Claude API Key（週報分析用） | [console.anthropic.com](https://console.anthropic.com/) |
| `GEMINI_API_KEY` | Google Gemini API Key（截圖視覺描述用，免費 tier） | [aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey)，格式為 `AIza...` |
| `THREADS_ACCESS_TOKEN` | Threads 長效 Access Token（60 天） | Meta Developer → App → Threads API |
| `THREADS_USER_ID` | 你的 Threads 用戶 ID | `curl "https://graph.threads.net/v1.0/me?fields=id&access_token=你的Token"` |
| `THREADS_TOKEN_REFRESHED_AT` | Token 最後刷新的 ISO 時間（供到期檢查用） | 本地執行後由程式自動寫入；首次需手動填入，例如 `2026-04-19T00:00:00.000Z` |

### Token 注意事項

Threads 預設給**短效 Token**（幾小時過期），必須先換成**長效 Token**（60 天）：

```bash
# 用短效 Token 換取長效 Token（需要 App ID 與 App Secret）
curl "https://graph.threads.net/access_token?grant_type=th_exchange_token&client_id=APP_ID&client_secret=APP_SECRET&access_token=短效Token"
```

長效 Token 每次執行 `npm run post` 時會**自動刷新並寫回 `.env`**，只要 60 天內執行過一次就不會過期。

## 發文輪替邏輯

### 文字貼文（`autoPost`）
依「今年第幾天」決定，不隨機，每天結果一致（可重現）：
- **功能**：8 個，以 `dayOfYear % 8` 輪替
- **風格**：8 種，以 `dayOfYear % POST_STYLES.length` 輪替（自動適應風格庫大小）
- **循環長度**：8 天後出現完全相同的組合（功能與風格數量相同時 LCM = 8）

### 圖片貼文（`imagePost`）
截圖與風格皆為**隨機選取**（`Math.random()`），每次執行結果不同：
- `--latest` 旗標可強制選 `screenshots.ts` 最後一筆（最新加入的截圖）
- 適合剛新增截圖時，確保新截圖優先被使用

## 排程設定（GitHub Actions）

已設定 5 個自動化 workflow，推送到 GitHub 後即啟用：

| Workflow 檔案 | 排程（台灣時間） | 執行內容 |
|---|---|---|
| `schedule-post.yml` | 每天 12:00 | `npm run post -- --yes`（長文） |
| `schedule-short-post.yml` | 每天 20:00 | `npm run post -- --yes --max-chars 130`（130 字以內短文） |
| `schedule-image-post.yml` | 每天 10:00 | `npm run image-post -- --yes` |
| `schedule-engage.yml` | 每天 21:00 | `npm run engage -- --yes`（需 `threads_manage_replies` 權限） |
| `schedule-report.yml` | 週一 09:00 | `npm run report -- --yes` + 建立 PR 等待人工 review |

### 啟用步驟

1. 前往 GitHub repo → **Settings → Secrets and variables → Actions**
2. 新增以下 Repository secrets：

| Secret 名稱 | 說明 |
|---|---|
| `HF_TOKEN` | Hugging Face Access Token |
| `ANTHROPIC_API_KEY` | Anthropic Claude API Key |
| `GEMINI_API_KEY` | Google Gemini API Key |
| `THREADS_ACCESS_TOKEN` | Threads 長效 Access Token（60 天） |
| `THREADS_USER_ID` | 你的 Threads 用戶 ID |
| `THREADS_TOKEN_REFRESHED_AT` | Token 最後刷新的 ISO 時間（供 `getUsableToken()` 計算剩餘天數） |

3. Secrets 設定完成後，Actions 會在下一個排程時間自動觸發
4. 也可至 **Actions** 頁面手動點 **Run workflow** 立即測試

### Token 到期注意

每次執行時，`getUsableToken()` 會顯示 Token 剩餘天數：
- 剩餘 > 5 天：正常顯示天數
- 剩餘 ≤ 5 天：自動嘗試刷新（本地執行會寫回 `.env`；GitHub Actions 無法寫回 Secrets，會輸出 `::warning::` 提示）
- `THREADS_TOKEN_REFRESHED_AT` 未設定：GitHub Actions 中輸出警告，建議設定此 Secret

Token 有效期 60 天，到期前請在本地執行任意指令（會自動刷新），並同步手動更新 `THREADS_ACCESS_TOKEN` 與 `THREADS_TOKEN_REFRESHED_AT` 兩個 Secret。

## 後續可擴充的方向

- **申請 `threads_manage_replies` 權限** — 取得後 `engage` 才能真正讀取自己貼文的留言（目前靜默略過）；申請路徑：Meta for Developers → App → Threads API → Request permission
- **Threads App Review（進階存取）** — 申請通過後 `reply-to` 可自動抓取他人公開貼文內容，不需手動複製貼上
- **`data/features.ts`** — 持續補充新功能描述；`imagePost.ts` 遇到新截圖時會自動擴充
- **`assets/screenshots/`** — 持續新增 SaaS 截圖（建議 10 張以上避免重複；檔名須為純 ASCII），commit 後加入 `screenshots.ts` 即可
- **週報 PR 改善** — 目前 PR body 提示查看 Files changed；可進一步在 body 直接嵌入 diff 摘要、自動關閉舊有 bot PR、拆分 reports 與 data 層更新為兩個步驟（見健診報告）
