# GlowPulse 專案健診報告

> **撰寫立場**：以外部技術顧問視角，對 codebase 進行中性、批判性的全面審視。  
> **目的**：找出真正的問題，而不是列優點清單。  
> **日期**：2026-04-19

---

## 一、執行摘要

GlowPulse 是一套為 GlowMoment SaaS 產品設計的 Threads 社群自動化工具。整體架構設計思路清晰，分層乾淨，自動化流程完整，屬於具備工程素養的個人專案。

**但以下五件事需要認真面對：**

1. **零測試覆蓋** — 整個 codebase 沒有任何自動化測試，AI 自動修改生產設定檔的機制完全無防護網
2. **AI 直寫生產設定** — `brand.ts`、`styles.ts` 會被 AI 自動覆寫，沒有審查機制，一次 bad output 就能悄悄破壞後續所有發文品質
3. **60 天定時炸彈** — Threads Token 過期後 GitHub Actions 會靜默失敗，沒有任何告警機制
4. **核心功能是殘缺的** — `engage` 任務因缺少 API 權限，實際上讀不到任何留言，但 Actions 排程仍每天跑
5. **單點設計** — 整套系統假設永遠只有一個 Threads 帳號，未來要擴充到多帳號或多平台會需要大規模重構

## 二、架構評估

### 2.1 值得肯定的設計

**分層清晰**，四層職責分離做得好：

```
agent.ts（CLI）→ tasks/（業務）→ services/（API）→ data/（設定）
```

- `data/` 層是純設定，沒有任何業務邏輯，非技術人員可以直接修改 `brand.ts` 調整語氣
- `generate()` 的 task-based model routing 設計很實用，切換模型只需改 `models.ts`，不碰任何業務代碼
- `GLOW_AUTO_CONFIRM=1` 機制讓本地互動模式與排程模式共用同一套代碼，沒有分叉

**錯誤處理架構合理**：
- services 層拋出例外
- tasks 層捕捉並回傳 `{ success: false, error }` 結構體
- agent.ts 統一處理退出碼

### 2.2 架構問題

#### 問題一：AI 直接寫入生產設定檔，沒有版本控管機制

`weeklyReport.ts` 和 `analyzeReference.ts` 的 AI 輸出會直接覆寫 `brand.ts` 與 `styles.ts`，並透過 GitHub Actions 自動 commit 到 main branch。

這個設計有根本性風險：

```
AI 輸出 bad JSON → parseDataUpdates() 成功解析（格式正確，但內容錯誤）
→ updateStylesTs() 覆寫掉所有精心調整的風格指令
→ 自動 commit 到 main
→ 之後所有發文品質靜默下降
→ 沒有任何人知道發生了什麼事
```

目前 `genStyles.ts` 已發生過一個具體案例：執行後 `styles.ts` 裡原本 3 個風格的 `instruction` 被 AI 改寫了（對比原始版本與現在的版本，內容明顯不同）。這不一定是壞事，但**完全沒有比對、沒有確認、沒有 diff 記錄**。

**真正的問題不是 AI 會寫錯，而是你不知道它寫了什麼。**

#### 問題二：自我學習迴路可能產生語意漂移

系統設計了一個閉環：

```
AI 生成貼文 → 發到 Threads → weeklyReport 分析互動數據
→ AI 更新 brand.ts 技巧清單 → 下次 AI 讀更新後的 brand.ts 生成貼文
```

這個設計理論上很優雅，實際上有個隱患：**AI 每次更新 brand.ts 是覆寫整個【從優質貼文學到的技巧】區塊，不是累積**。如果某次週報的互動樣本偏少（例如：當週只發了 2 篇貼文），AI 可能用不具代表性的樣本覆寫掉前幾週累積的有效技巧。

目前沒有機制可以「回滾」到上個版本的 brand.ts。

#### 問題三：styles.ts 的第一條到第三條每次都可能被 AI 改寫

`analyzeReference.ts` 的 AI prompt 要求輸出完整的 styles 陣列（包含現有 3 個），這表示 AI 不只「新增」風格，也會「修改」現有風格的 instruction。而 genStyles 的 prompt 也有同樣設計。

**人工精心調整的風格指令，在每次 learn / gen-styles 後都有被覆蓋的風險。**

## 三、功能逐項評估

### 3.1 autoPost — 良好，有一個小 bug

**運作正常**，日期輪替邏輯（`dayOfYear % POST_STYLES.length`）確保同天重跑結果一致，是好設計。

**一個需要注意的問題**：
溢出處理的「以留言接續」邏輯依賴 `waitForPostReady()`，最多等 30 秒。如果 Threads 後端當天特別慢（偶發），主貼文會超時失敗，整個流程中止。目前沒有 retry 機制。

**另一個觀察**：
字數計算用的是 JavaScript `string.length`，這對 emoji 是有問題的。`🪄` 的 `.length` 是 2（UTF-16 代理對），但 Threads 通常計算為 1 字。如果貼文包含大量 emoji，實際顯示字數會比 `maxChars` 少，表示字數限制過於嚴格。

### 3.2 imagePost — 架構上有斷點

Gemini 視覺分析是 imagePost 的唯一入口，但它**沒有 fallback**：

```typescript
// gemini.ts
export async function describeImageWithGemini(imagePath: string): Promise<string>
// 失敗直接拋出 Error，沒有 try fallback
```

只要 Gemini 配額耗盡或 API 有問題，整個 imagePost 功能就完全停止。而 GitHub Actions 會靜默失敗（沒有 Slack 通知或任何告警）。

**另外**：`screenshots.ts` 目前只有 3 個條目。「隨機選取」的效果是每幾天同一張圖就會重複出現，頻率頗高。

### 3.3 engage — 功能是殘缺的

這是目前最嚴重的問題。`getPostReplies()` 需要 `threads_manage_replies` 或 `threads_read_replies` 進階權限，而根據對話記錄，這個權限**尚未申請到**。

實際運作結果：
```
掃描貼文 → getPostReplies() 返回空陣列（無錯誤，靜默失敗）
→ totalAnalyzed = 0
→ "沒有需要回覆的留言。"
→ 每天 21:00 GitHub Actions 跑完，什麼都沒發生
```

這個 Actions 每天消耗一次執行配額，但不做任何有意義的事情。

**建議**：在 `getPostReplies()` 加入權限檢查，若返回 403 則明確報錯；在功能實際可用之前，暫停 `schedule-engage.yml`。

### 3.4 weeklyReport — 邏輯正確，但資料品質有疑慮

計分公式 `likes×3 + reposts×5 + quotes×4`（排除 replies 和 views）邏輯合理。

**一個結構性問題**：
帳號剛起步，每週貼文量少，週報的「高互動 5 篇 vs 低互動 5 篇」的差異可能不具統計意義。如果某週所有貼文互動都很低，AI 分析出來的「高互動特徵」其實只是「稍微沒那麼差」的貼文，據此更新 brand.ts 可能是錯誤的訊號。

建議：週報在貼文數量 < 10 篇時，應跳過 AI 分析，只輸出數據，等樣本量夠再做建議。

### 3.5 genStyles — 最新功能，有個設計矛盾

`genStyles.ts` 的 prompt 要求 AI 輸出「完整的 styles 陣列」（包含原有 3 個），但實際執行後原有風格的 instruction 有被改寫（已驗證）。

這個行為模糊了一個重要邊界：**這個指令是「新增風格」還是「重寫所有風格」？**

如果目標只是新增，prompt 應要求 AI 只輸出新增的條目，然後由代碼在 `updateStylesTs` 之前合併（`[...POST_STYLES, ...newStyles]`），而不是讓 AI 決定舊風格要不要改。

### 3.6 零樣本分類（HF BART）— 準確度未知

`callZeroShot()` 使用 `facebook/bart-large-mnli` 進行四分類：
`['預約需求', '美容抱怨', '日常生活', '商業廣告']`

這個模型是英文訓練的，分類繁體中文時的準確度是未知的。目前沒有任何 ground truth 資料集或準確度評估。

**潛在後果**：
- `engage` 功能即使拿到了留言，分類結果可能大量誤判
- 真正的潛在客戶被分類為「日常生活」而略過
- 廣告留言被分類為「美容抱怨」而觸發回覆

## 五、技術負債評估

### 5.1 零測試覆蓋（最高優先）

整個 codebase 完全沒有測試。這在 MVP 階段可以接受，但現在系統已在生產環境自動運行，風險級別已不同。

最需要測試的三個模組：

**① `dataUpdater.ts`** — 這個模組直接修改生產設定檔，一個 bug 會破壞所有後續發文：
- `updateBrandTs()` 的正則邊界條件
- `updateStylesTs()` 的字串跳脫（單引號、反斜線、換行）
- `parseDataUpdates()` 的 JSON 容錯

**② `publish.ts` 的 splitContent()**  — 文字分割邏輯有多個優先順序（段落→句尾→換行→硬截），邊界條件難以靠人工驗證。

**③ `threads.ts` 的 `parseThreadsPostId()`** — Base64URL 解碼邏輯，各種 URL 格式都要測到。

### 5.2 沒有 Dry Run 模式

現在測試任何功能都必須實際呼叫 Threads API 並發文。這導致：
- 測試代碼改動時有意外發文風險
- 無法在本地快速驗證邏輯
- GitHub Actions 的 `workflow_dispatch` 等同於「直接推到生產」

建議增加 `--dry-run` 旗標，讓所有任務在不實際呼叫 Threads 的情況下跑完整個流程，輸出「如果真的發布，內容會是這樣」。

### 5.3 設定檔是可執行代碼（風險被低估）

`brand.ts`、`styles.ts`、`features.ts` 雖然看起來像設定檔，但它們是 TypeScript 模組。AI 自動更新時用的是字串替換（`writeFileSync`），如果 AI 生成的 instruction 裡面包含未跳脫的單引號，`updateStylesTs` 會產生語法錯誤的 TypeScript 檔案。

```typescript
// 這種情況會破壞 styles.ts：
instruction: 'AI 可能輸出 it\'s a trap 這樣的文字',
//                              ↑ 跳脫邏輯如果有 bug，這裡就語法錯誤
```

目前的跳脫邏輯：
```typescript
.replace(/\\/g, '\\\\')
.replace(/'/g, "\\'")
.replace(/\r?\n/g, ' ')
```

這是正確的，但沒有測試覆蓋，萬一有漏洞，下次執行 `tsx` 就會報 SyntaxError，整個服務中斷。

## 六、產品策略觀察

### 6.1 發文量與品質的矛盾

目前排程：
- 週一/三/五：長文（`autoPost`）
- 週二/四：圖文（`imagePost`）
- 每天 20:00：短文（130 字內）

每天至少一篇，週間最多兩篇。這個頻率對於起步帳號是合理的。

**但有一個潛在問題**：短文(`--max-chars 130`) 和長文使用完全相同的 `autoPost` 邏輯和 `BRAND_CONTEXT`，只是加了字數限制。130 字內的短文需要更強的「開頭鉤子」和「留言誘導」，但目前沒有為短文設計專屬的 prompt 策略。

`brand.ts` 的【誘導留言格式】只是附加資訊，AI 不一定在短文裡真的使用。建議為短文建立專屬的 PostStyle 或在短文 prompt 中強制要求使用誘導留言結尾。

### 6.2 8 個風格輪替的問題

`autoPost` 用 `dayOfYear % POST_STYLES.length` 輪替。現在有 8 個風格，配合 8 個功能，理論上每 8 天一個完整循環。

**問題**：新增的 5 個風格（問題解惑型、情境應用型、故事揭秘型、趨勢觀察型、資訊引導型）是 AI 自動生成的，**品質未驗證**。建議在正式排程中跑之前，手動用 `npm run post` 預覽各風格實際生成的貼文，確認品質達到標準再讓它排程自動運作。

### 6.3 截圖更新頻率嚴重不足

`screenshots.ts` 目前只有 3 個條目。imagePost 「隨機選取」在只有 3 個選項的情況下，大約每 3 天就會重複同一張圖。Threads 演算法可能對重複發相同圖片的帳號降低觸及。

建議：至少準備 10-15 張不同功能的截圖，確保同一張圖不會在 2 週內重複出現。

## 七、改進建議（優先順序排列）

### 🔴 高優先（應立即處理）

**① 為 AI 自動更新 data 層加入保護機制**

最小成本方案：自動 commit 改為建立 PR（`gh pr create`），人工 review 後才 merge。

✅ **已實作（2026-04-19）**：`schedule-report.yml` 改為建立 `bot/weekly-report-YYYY-MM-DD` branch 並開 PR，加入 `pull-requests: write` 權限，PR body 提醒查看 Files changed tab。

**後續可改善的方向**：

- **PR body 自動貼入 diff 摘要**：目前 PR 說「請看 Files changed」，更好的做法是把 `brand.ts` / `styles.ts` 的新舊差異直接貼在 PR body 裡，不用點進去就能判斷要不要 merge。
- **自動關閉舊有 bot PR**：若上週 PR 未 merge，本週又開了一個，倉庫裡會同時有兩個 PR。可在建新 PR 前先查找並關閉舊的：
  ```bash
  gh pr list --label bot --state open --json number -q '.[].number' | xargs -I{} gh pr close {}
  ```
- **分開處理**：`docs/reports/` 週報文件直接 commit main 沒問題（只是紀錄），只有 `brand.ts` / `styles.ts` 的改動需要人工審查。可以把兩件事拆成兩個步驟，reports 直接推，data 層才開 PR，減少每週要 review 的內容量。
- **大改動標示**：如果 AI 改動超過 N 行，在 PR title 自動加上 ⚠️，讓你第一眼就知道這次改動幅度較大。

**② 為 Threads Token 過期加入告警**

在每次執行時，計算 Token 剩餘天數。若剩餘 ≤ 5 天，在 GitHub Actions log 中輸出顯眼警告。

✅ **已實作（2026-04-19）**：`getUsableToken()` 現在每次執行都顯示剩餘天數；≤ 5 天自動刷新並輸出 `::warning::` annotation；`THREADS_TOKEN_REFRESHED_AT` 已加入所有五個 workflow 的 env。

**使用前需要做的一步**：在 GitHub Repository Secrets 新增 `THREADS_TOKEN_REFRESHED_AT`，值填入你當初取得長效 Token 的 ISO 時間（例如 `2026-03-01T00:00:00.000Z`）。之後每次本地執行都會自動刷新並更新 `.env`；GitHub Actions 環境因無法自動寫回 Secrets，到期時會顯示警告，需手動更新兩個 Secret（`THREADS_ACCESS_TOKEN` 和 `THREADS_TOKEN_REFRESHED_AT`）。

**③ 暫停 `schedule-engage.yml` 直到權限申請成功**

每天空跑沒有意義，浪費 GitHub Actions 配額，建議先 disable。

**關於 `threads_manage_replies` 權限**：讀取留言（即使是自己的貼文）需要申請此權限，Meta 將「讀留言」和「發留言」分開管理。目前狀況：
- 發布回覆（`threads_content_publish`）✅ 已有
- 讀取留言（`threads_manage_replies`）❌ 需申請

申請步驟：Meta for Developers → 你的 App → Threads API → 找到 `threads_manage_replies` → Request → 填寫使用說明（說明用途是讀取自己貼文的留言做客服回覆）→ 等 App Review 審核（通常數天）。

### 🟡 中優先（近期規劃）

**④ 為 genStyles 修正「只新增不修改舊有」的設計**

```typescript
// genStyles.ts 應改為：
// AI 只輸出新風格，然後由代碼合併
const allStyles = [...POST_STYLES, ...newStylesFromAI];
updateStylesTs(allStyles);
```

修改 prompt：移除「請保留原有三個風格」的指令，改為「只輸出全新的風格條目，不要包含職人共鳴感、創業乾貨型、視覺至上型」。

**⑤ 增加 `--dry-run` 模式**

所有任務加入 dry run 支援：跑完整流程但不實際呼叫 Threads 發布 API，只輸出「預計發布內容」。

**⑥ 增加截圖數量至 10+ 張**

補充 `screenshots.ts`，確保圖片貼文不會頻繁重複。

**⑦ 在 weeklyReport 加入最小樣本量保護**

```typescript
if (myPosts.length < 10) {
    console.log('⚠️  本週貼文數量不足（${myPosts.length} 篇），不進行 AI 分析，避免樣本偏差');
    // 只輸出數據，不更新 brand.ts / styles.ts
    return;
}
```

### 🟢 長期規劃

**⑧ 為核心模組補充測試**

最低限度：`dataUpdater.ts`（updateStylesTs 字串跳脫邏輯）、`publish.ts`（splitContent 分割邏輯）、`threads.ts`（parseThreadsPostId URL 解析）。

不需要完整測試套件，針對高風險函式寫 10-20 個 unit test 就能大幅降低風險。

**⑨ 評估零樣本分類準確度**

收集 50-100 筆真實留言樣本，人工標記分類，計算 `facebook/bart-large-mnli` 在繁體中文的 precision/recall。如果低於 70%，考慮改用 Gemini 或微調過的繁體中文模型。

**⑩ 短文專屬 PostStyle**

為 `--max-chars 130` 的短文建立 1-2 個專屬風格，強制使用誘導留言格式（問句型、二擇一、投票感），提升短文互動率。

## 八、總結

GlowPulse 是一個**架構設計優於工程穩健性**的專案。分層設計、模型路由、品牌知識庫分離等概念都做得很好，這是少見的。

但它目前的最大問題是：**一個自動修改自己的系統，但沒有任何機制讓你知道它改了什麼**。

AI 自動更新 `brand.ts` 和 `styles.ts` 是這套系統最有價值的設計，也是最危險的設計。在加入任何形式的審查機制（哪怕只是 PR）之前，它就像一個會自我學習但沒有記憶的系統——它今天的改動，你三週後才可能察覺。

以目前的開發資源（推測為一人團隊），建議按本報告的優先順序處理，先解決「高優先」的三個問題，可以在半天內完成，但能大幅降低最主要的運維風險。
